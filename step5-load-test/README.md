## 핵심 개념

### Redis가 빠른 이유

```
1. 인메모리: 디스크(~10ms) → 메모리(~100ns) → 10만 배 빠름
2. Single-thread 이벤트 루프: 컨텍스트 스위칭 없음, 락 없음
3. epoll I/O Multiplexing: 수천 연결을 스레드 1개로 처리
4. 단순한 자료구조: 복잡한 연산 없음

결과: ~10만 ops/sec
```

### Virtual Thread (Java 21+)

```
플랫폼 스레드:
  Redis 응답 대기 중 → OS 스레드 점유
  스레드 200개 → 동시에 200개 요청만 처리 가능

Virtual Thread:
  Redis 응답 대기 중 → OS 스레드 반환 (다른 작업에 양보)
  수만 개 생성 가능 → 더 많은 동시 요청 처리
```

### TPS 공식

```
TPS = 요청 수 / 소요 시간(초)
예: 10,000 요청 / 0.743초 = 13,458 TPS
```

### 실무 병목 지점

```
Redis 자체:    ~10만 ops/sec (Single-thread 한계)
클라이언트:   스레드 부족, 연결 풀 부족 → 대기 적체
네트워크:     RTT 누적

→ Redis보다 클라이언트 / 네트워크가 먼저 병목
→ Virtual Thread + 연결 풀 최적화로 개선
```
